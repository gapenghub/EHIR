MOD 02

Footprinting & Reconnaissance

passive vs active · DNS/AXFR · social eng

Passive vs Active footprinting EXAM ★ ESSAY 1

Prep especially wants passive recon: gathering from public sources with no contact with the target.

Passive

No direct interaction → stealthy, no requests reach the target. OSINT, search engines, social media, job posts, Whois, DNS records, public repos (GitHub/Pastebin), document metadata.

Active

Directly touches the target → detectable, but returns more detail. Port/network scanning, service fingerprinting, email harvesting, website/directory enumeration, traceroute, and DNS zone transfer (AXFR). Tool the slide names: Nmap.

Methodology order: identify target → passive → active → network info → social engineering → website → DNS → compile & report.

Two easy marks: use passive first to avoid tripping the IDS/IPS, and get authorisation before any active scanning (the legal/ethical line).

Essay tip — don't write the same thing twice

Your Module 02 slide files port/network scanning under active footprinting, yet there's also a whole Scanning module (MOD 03) — they overlap on purpose. In Essay 1 keep active recon at footprinting level (traceroute, zone transfer, email/website enumeration, an initial host probe); save the deep scan mechanics — SYN vs Connect, NULL/FIN/Xmas, IDS evasion — for Essay 2.

DNS zone transfer / AXFR EXAM ★

A secondary DNS server normally requests a full copy of the zone from the primary using an AXFR request (for redundancy). If the primary is misconfigured to allow anyone, an attacker mimics a secondary and pulls the entire zone file — every hostname, IP and record in one shot.

What the attacker gains: a full internal map — subdomains, mail servers, internal IPs, key hosts to target next.

# test for an open zone transfer
dig axfr @ns1.example.com example.com
nslookup -type=axfr example.com ns1.example.com

Social engineering — focus: phishing EXAM

Manipulating people into revealing info or creds — exploits human weakness, not a system bug.

Techniques

  • Phishing — fake emails harvesting credentials.
  • Impersonation — pose as IT/vendor.
  • Dumpster diving — secrets in the trash.
  • Shoulder surfing / eavesdropping.
  • Social-media profiling — habits & targets from LinkedIn/FB/IG.

Mitigation & limits

Mitigate: security-awareness training, a reporting culture, verification procedures, minimise public info. Limitation: phishing relies on human error — trained, skeptical staff and reporting defeat it; hard to target precisely at scale.

MOD 03

Computer & Network Scanning

★ Heavy zone — also Essay 2

TCP 3-way handshake & flags EXAM ★

Setup: SYN → SYN/ACK → ACK = connection established. Teardown: graceful FIN exchange, or abrupt RST.

FlagMeaning
SYNSynchronize — start a connection
ACKAcknowledge a received segment
FINFinish — graceful close
RSTReset — abort the connection
PSHPush — deliver data immediately
URGUrgent — prioritise this data

Host discovery: ARP vs ICMP EXAM ★ ESSAY 2

Finding live hosts (different question from finding open ports).

  • ICMP echo ping (-PE) — classic ping; simple but often blocked by firewalls.
  • ARP ping scan — Layer-2; most reliable on the local subnet, finds hosts even behind restrictive firewalls, and shows the MAC address. Can't cross routers.
  • UDP ping (-PU, default port 40125) — useful against hosts with strict TCP filtering.
# ping sweep + verify
nmap -sn 192.168.x.0/24 # host discovery only, no port scan
arp-scan --localnet

Port scanning techniques EXAM ESSAY 2

ScanNmapHow / why
TCP Connect (full open)-sTCompletes the full handshake → reliable but logged.
SYN / half-open (stealth)-sSSYN→SYN/ACK→RST (never finishes). Stealthier, needs root.
NULL / FIN / Xmas-sN -sF -sXInverse flag scans. Open port = no reply, closed = RST.
UDP-sUNo handshake; slow; result often open|filtered.
ACK-sAMaps firewall rules (stateful vs simple filter).

TRAP Inverse scans (NULL/FIN/Xmas): FIN=FIN only, NULL=no flags, Xmas=FIN+URG+PSH. Work on BSD-style stacks, not on Windows hosts.

Service/version detection & banner grabbing EXAM ★

nmap -sV fingerprints the service + version; nmap -O guesses the OS. Why it matters: a version string lets you map known CVEs straight to that service and pick a matching exploit.

Active banner grab

Send crafted/malformed packets; compare the TCP/IP-stack responses to a database.

Passive banner grab

Sniff existing traffic and infer the OS from telltale signs — no probing.

Scanning beyond IDS/firewall (evasion) EXAM ★ ESSAY 2

Packet fragmentation — split the probe into tiny fragments; many IDS skip fragmented packets (too costly to reassemble), and the host reassembles them. Tools: nmap -f, fragroute.

The rest of the evasion toolkit

  • IP decoys (-D) & IP spoofing — hide your real source among fakes.
  • Source-port manipulation — disguise as a trusted port (e.g. 53/80).
  • Source routing, custom packets, bad checksums.
  • Randomize host order; slow / timing scans to stay under detection thresholds.
  • Proxy servers / proxy chaining & anonymizers — bury the origin.

MOD 06

System Hacking & Post-Exploitation

PrivEsc · persistence · C2 · covering tracks

Privilege escalation: horizontal vs vertical EXAM ★

Horizontal

Same privilege level, but access another peer user's stuff. E.g. banking user A reaches user B's account.

Vertical

Jump to a higher privilege — user → admin/root. E.g. a normal user reaching admin functions.

Gaining & maintaining access EXAM ESSAY 1

Gain access: crack passwords, exploit a buffer overflow, or exploit a known vulnerability (often via Metasploit). Maintain access (persistence): install backdoors / rootkits, hide files via NTFS ADS, keep a reverse shell.

Malware C2 behaviour EXAM

A compromised host beacons / calls back to an attacker-controlled server over a two-way channel — web traffic, email, or DNS — usually encrypted to hide it. Network anti-forensics channels: reverse HTTP shells, reverse ICMP tunnels, DNS tunneling.

Covering tracks / anti-forensics EXAM ★ ESSAY 1

  • Disable auditing, then clear or manipulate logs (delete only attack events to dodge detection).
  • Windows event logs: SECEVENT.EVT (security — failed logins, file access without rights), SYSEVENT.EVT (system — driver failures), APPEVENT.EVT (applications).
  • Tooling: clearev in a Meterpreter shell; wevtutil; PowerShell; the Clear_Event_Viewer_Logs.bat utility.
  • Hide on OS via NTFS ADS; delete unrecoverably with cipher.exe; disable last-access timestamps / restore points.

Authentication log analysis EXAM

Defender side: tell successful vs failed logins apart. A burst of failures then a success = brute force / password guessing. Watch for off-hours admin logins = privilege abuse.

MOD 07

Incident Response & Digital Forensics

IR life cycle · evidence · chain of custody

★ The 4-vs-6-phase trap

Your prep flags this. Three models float around — know which your question wants. Default to the NIST 4-phase (what the Module 07 lecture teaches); be ready to mention the others.

  • NIST 4-phase (lecture): Preparation → Detection & Analysis → Containment, Eradication & Recovery → Post-Incident Activity (Lessons Learned).
  • SANS 6-step: Preparation → Identification → Containment → Eradication → Recovery → Lessons Learned. (This is the one your group forensic assignment used — and Essay 3 mirrors that assignment.)
  • EC-Council IH&R 9-step (from Module 01) — the granular version.

IR goals & detection EXAM ESSAY 3

Goals: minimise damage, reduce recovery time, restore normal operations fast. Detection & Analysis is the hardest phase — "incidents can't always be prevented but must always be detected."

Precursors vs Indicators

Precursor

A sign an incident may happen — e.g. a vuln scanner in your web logs, a public exploit released for your mail server.

Indicator

A sign one has/ is happening — e.g. AV flags malware, IDS alerts on a successful DB attack.

Common attack vectors

External/removable media (USB), Attrition (brute force), Web, Email, Impersonation (spoofing/MITM/rogue AP/SQLi), Improper usage, Loss/theft of equipment.

Evidence gathering & handling EXAM ★ ESSAY 3

Keep a detailed log for every piece of evidence — this is the basis of chain of custody:

  • Identifying info: location, serial & model number, hostname, MAC & IP.
  • Name, title & phone of everyone who collected or handled it.
  • Time & date (with time zone) of every handling event.
  • Where it was stored (tamper-evident, secure).

Containment strategy is chosen on: potential damage, need to preserve evidence, service availability, time/resources, and effectiveness.

Eradication, recovery & lessons learned EXAM

Eradicate: delete malware, disable breached accounts, fix exploited vulns across all affected hosts. Recover: restore from clean backups, rebuild, patch, change passwords, tighten the perimeter. Lessons learned (most often skipped): what happened, when, how well the team did, what to watch for next time.

MOD 08

Memory Acquisition & Analysis

imaging · hashing · Volatility

Forensics process & hashing EXAM ★ ESSAY 3

Process: Acquisition → Analysis → Documentation/Presentation. Goals: what happened, how, who, when, where, why.

Imaging = a bit-for-bit forensic copy (use a write blocker; work on the copy, preserve the original). Hashing proves integrity: take an MD5/SHA-256 hash before and after — if the hash is unchanged, the evidence is unaltered.

TRAP Hashing = integrity, not confidentiality, and it's one-way (can't be reversed).

Why memory is gold EXAM

Volatile RAM holds things the disk never sees: running processes, executed commands, cleartext passwords, unencrypted data, open network connections, IPs, injected trojans. Capture it before powering off.

Volatility workflow ESSAY 3

# identify, then list, then hunt
vol.py -f mem.raw imageinfo
vol.py -f mem.raw --profile=<P> pslist / pstree / psscan
vol.py -f mem.raw --profile=<P> psxview # find hidden processes
vol.py -f mem.raw --profile=<P> netscan # network connections

IoC A great red flag: lsass.exe's parent should be wininit.exe. If its parent is explorer.exe, it's a fake process wearing a legit name.

MOD 10

Cryptography Bonus · light in prep

symmetric · asymmetric · hashing

Your prep only touches crypto through hashing for evidence integrity (covered in Module 08). The rest is here as MCQ insurance — if your lecturer confirmed crypto is out of scope, hit "Exam targets only" to hide it.

Symmetric vs asymmetric

Symmetric (private-key)

Same key encrypts & decrypts. Fast, for bulk data. DES, AES, RC4, IDEA.

Asymmetric (public-key)

Two keys (public + private). RSA, Diffie-Hellman, El Gamal, DSA.

Block vs stream: block ciphers work on fixed-size blocks (AES, DES — most modern ciphers); stream ciphers encrypt bit-by-bit with a keystream (RC4, SEAL).

Hashing (one-way) links Essay 3

MD5, SHA-1, SHA-2 produce a fixed-size digest that can't be reversed. Used to prove a file/evidence hasn't changed. The 4 crypto objectives: confidentiality, integrity, authentication, non-repudiation.

Bonus · Diffie-Hellman

Key exchange over an open line

Agree on public primes n, g; each side picks a secret and computes A=g^x mod n / B=g^y mod n; both derive the same shared key. Lab note: it's vulnerable to a man-in-the-middle attack — that's a likely essay/lab question.

MOD 09

Steganography Bonus · not in prep

covert channels · LSB · ADS

Not flagged anywhere in your prep — pure bonus. Worth a skim for MCQs (it also connects to "hiding files" in Module 06).

The core idea

Steganography = hiding the very existence of a message (vs cryptography, which hides the meaning). It's one kind of covert channel — moving data between parties who aren't supposed to communicate, slipping past firewalls/IDS.

Types: image, audio, text, email, and whitespace (trailing spaces/tabs).

LSB & ADS

Least Significant Bit — swap the rightmost bit of each byte/pixel for your secret bits; the image looks unchanged. Alternate Data Streams (NTFS) — hide data in a second stream: normal.txt:hidden.txt doesn't show in Explorer.

Fun fact · history

Old-school hiding

Demaratus hid a message under the wax of a tablet; WWII used invisible ink and the null cipher — read the first letter of each word: "Please Do Not Throw Sausage Pizza Away."

Fun fact · malware

Stegoloader

A real trojan that hid its main module inside a PNG image downloaded from a legit site to dodge detection.

Bonus · detection

File signatures & tools

Magic numbers (file header) reveal the true type even if the extension lies. Tools: TrID, Binwalk, file; stego tools: Steghide, QuickStego, MP3Stegz, SNOW.

Attack life cycle & methodology

Modules 02·03·04·06›

What this question tests. Whether you can walk an attacker's full path against the organisation in the scenario, name the correct technique at each stage, explain why each stage depends on the one before it, and pair the attack with a defence.

The attack lifecycle, stage by stage

Two models describe the same path at different levels of detail. The Certified Ethical Hacker five phases are Reconnaissance → Scanning → Gaining Access → Maintaining Access → Clearing Tracks. The Lockheed Martin Cyber Kill Chain breaks it into seven: Reconnaissance → Weaponization → Delivery → Exploitation → Installation → Command and Control → Actions on Objectives. Use whichever fits the question, but the marks come from showing how each stage hands something to the next:

  • Reconnaissance — gather information about the target. Passive reconnaissance comes first because it makes no contact with the target and so leaves no trace: Whois records, Domain Name System (DNS) records, certificate-transparency logs (crt.sh) for subdomains, Google search operators, email harvesting with theHarvester, employee details from LinkedIn and job adverts, and metadata inside published documents. Active reconnaissance then touches the target directly to fill in the gaps.
  • Scanning — turn that information into a map of the live network. Host discovery (an Address Resolution Protocol or Internet Control Message Protocol ping sweep) answers which machines are switched on; a port and service scan with Nmap then shows which services are listening and which versions they are running. Keep this at summary level here — the stealth scans, flag types and IDS evasion belong to Essay 2, so naming the step is enough.
  • Enumeration — make active connections to the open services and pull out specifics: usernames, machine names, shared folders, and software versions. This is the stage that converts a bare fact like "port 445 is open" into something usable such as "Server Message Block version 1, Windows 2000, share named ACCOUNTS".
  • Gaining access — take those exact version details, match them to a known vulnerability (a Common Vulnerabilities and Exposures, or CVE, identifier), and run the matching exploit. Without the version from enumeration you would only be guessing, which is why the earlier stages matter.
  • Maintaining access — a single session is lost the moment the machine reboots, so the attacker raises their privilege level (privilege escalation) and installs something that survives a restart: a backdoor, a rootkit, or a scheduled task.
  • Clearing tracks — finally the attacker removes the evidence so the defender cannot reconstruct what happened: disabling auditing, clearing or selectively editing the event logs, and altering file timestamps.

Defences to pair with each stage

If the question asks how the organisation could have prevented the attack, give a control for each stage rather than one blanket answer: minimise public information and monitor web logs (reconnaissance); patch and harden systems (weaponization and exploitation); filter email and train staff to recognise phishing (delivery); use endpoint protection, application whitelisting and file-integrity monitoring to catch installed malware (installation); segment the network and apply egress filtering to break command-and-control traffic; and keep an incident-response plan and clean backups for the final stage. One easy mark people miss: as an ethical hacker you would only carry any of this out within an authorised, written scope of engagement.

Worked example — a company running legacy systems

Reconnaissance through Whois and certificate-transparency logs reveals an old server, and a job advert mentions "Windows Server 2000". A service-version scan confirms that the host has Server Message Block listening on port 445. Enumeration through a null session lists its shares and user accounts. The legacy Server Message Block service is vulnerable to MS08-067 (CVE-2008-4250), a critical remote-code-execution flaw, so the attacker runs that exploit and obtains a session. They then escalate privilege, install a backdoor for persistence, and clear the logs. The defence is to patch or decommission the end-of-life operating system, disable Server Message Block version 1, segment the host away from the main network, apply least privilege, and filter inbound email to close the phishing route.

Common mistakes

  • Listing the phases as generic theory with no link to the organisation in the scenario.
  • Forgetting the defensive half when the question explicitly asks how the attack could be prevented.
  • Naming a stage but not explaining how it feeds the next one — that explanation is where the marks sit.

Scanning, host/port discovery & IDS evasion

Module 03›

What this question tests. Whether you understand the difference between finding which hosts are alive and finding which services are open, whether you can explain the packets behind each scan, and whether you can explain how unusual or crafted packets get past an Intrusion Detection System or firewall.

Two different questions: live hosts versus open services

These are separate steps and the examiner wants them kept separate. Host discovery answers "which machines are switched on". An Internet Control Message Protocol echo request (the ordinary ping) is the simplest method but is often blocked by firewalls. An Address Resolution Protocol scan works at the data-link layer, so it is very reliable on the local subnet and even reveals each machine's hardware (MAC) address, but it cannot cross a router to reach another network. Port and service discovery answers the different question of "which doors are open and what is behind them", which you find by scanning for open ports and then fingerprinting the service and its version.

The scan types, and why they behave that way

  • TCP Connect scan completes the full three-way handshake (SYN, then SYN/ACK, then ACK), so it is reliable but it is also recorded — a completed connection appears in the target's logs.
  • SYN (half-open) scan is the stealthy alternative: it sends a SYN, receives the SYN/ACK, then replies with a RST instead of an ACK, so the connection is never actually established. Older logging that only records established connections never sees it.
  • NULL, FIN and Xmas scans send unusual flag combinations. By the Transmission Control Protocol standard, a closed port replies with a RST while an open port stays silent, so "no reply" indicates an open port. The catch worth stating: these work on systems with a standard (BSD-derived) network stack but not on Windows, which replies with a RST regardless and breaks the technique.
  • Service and version detection (the Nmap -sV option) identifies the software and its version, and operating-system detection (-O) guesses the platform. This matters because a version string is what lets you look up a known vulnerability and choose a matching exploit — it is the whole reason for scanning before attacking.

Getting past an Intrusion Detection System or firewall

Explain the mechanism for each technique rather than just naming it:

  • Packet fragmentation splits the probe into very small fragments. Many detection systems will not pay the cost of reassembling fragments, so the attack signature never matches, while the target host reassembles them and processes the probe normally.
  • Decoy addresses and source-address spoofing hide your real source by surrounding it with fake ones, so an analyst cannot tell which address is genuinely scanning.
  • Slow or timed scans spread the probes out over time to stay underneath the rate threshold that the detection system uses to recognise a scan.
  • Source-port manipulation sends traffic from a trusted port such as 53 or 80 to slip through firewall rules that permit that kind of traffic.
  • Proxies, proxy chaining and anonymizers route the traffic through intermediaries so that any logs point at them instead of at you.

Worked example — a firewall that blocks pings

Because Internet Control Message Protocol pings return nothing, you would discover live hosts with an Address Resolution Protocol scan on the local subnet, which is reliable even behind a restrictive firewall. On each host you would run a SYN scan for stealth together with service-version detection to identify the running software. To get past the detection system you would combine fragmentation with a few decoy addresses and a slow timing template: fragments avoid signature matching, decoys hide your address, and the slow timing stays under the scan-rate alarm. The services and versions you find then feed enumeration so you can choose the exploit.

Common mistakes

  • Treating host discovery and port scanning as the same thing — they answer different questions.
  • Forgetting that NULL, FIN and Xmas scans do not work against Windows hosts.
  • Filing evasion under enumeration — it belongs to the scanning module.

Incident response & digital evidence handling

Modules 07 & 08 · mirrors your group assignment›

What this question tests. Whether you can run an incident from start to finish using a recognised response lifecycle, and whether you can handle digital evidence so that it stays admissible in court. This is the essay that mirrors your group forensic assignment, so draw on what you did there.

Choose a response lifecycle

Three frameworks exist and they are the same process at different levels of detail — showing that you know this is itself a mark. The National Institute of Standards and Technology (NIST) four-phase model is Preparation → Detection and Analysis → Containment, Eradication and Recovery → Post-Incident Activity. The SANS six-step model is Preparation → Identification → Containment → Eradication → Recovery → Lessons Learned; it simply pulls Identification out and splits the combined middle phase into separate steps. The EC-Council nine-step Incident Handling and Response process breaks it down further still (Preparation, Recording and Assignment, Triage, Notification, Containment, Evidence Gathering and Forensics, Eradication, Recovery, Post-Incident Activities). Use the four-phase model as your backbone unless the question names a particular one.

Spotting the incident: precursors versus indicators

Detection and Analysis is the hardest phase because you have to recognise the incident from weak signals, and the two terms have distinct meanings. A precursor is a sign that an incident may be about to happen — for example a vulnerability scanner appearing in your web logs, or a new public exploit being released for software you run. An indicator is a sign that an incident is already happening or has happened — for example antivirus flagging malware, an Intrusion Detection System alerting on a successful attack, or a file-integrity tool reporting that a system file has changed.

Handling the evidence — the highest-value part of this essay

Four practices keep evidence admissible. Explain why each one matters, not just its name:

  • A bit-for-bit forensic image taken through a write blocker. You copy the drive exactly, which captures slack space and deleted data rather than only the visible files, and you always work on the copy. The write blocker physically prevents any write to the original drive, because a single write changes the disk and lets the opposing side argue the evidence was altered.
  • The order of volatility. You capture the most volatile data first — the contents of memory (RAM) before you power the machine off — because memory holds running processes, executed commands, passwords in cleartext, live network connections and injected malware, all of which disappear the instant the system is shut down.
  • Hashing for integrity. You calculate a hash value (using Message Digest 5 or Secure Hash Algorithm 256) of the image before and after handling it. If the two hash values match, you have mathematically proven that the data was not changed; this demonstrates integrity. Hashing is one-way and is not the same as encryption.
  • Chain of custody. You keep a documented log of who handled each item, when (including the time zone), and where, for every handover, and you store the evidence in secure, tamper-evident conditions. This removes any argument that someone could have tampered with the evidence between collection and court.

If the scenario involves a memory image, the analysis tool is the Volatility Framework: identify the image profile with imageinfo, list processes with pslist and pstree, reveal hidden processes with psxview, and list network connections with netscan. A reliable warning sign is a process named lsass.exe whose parent process is explorer.exe rather than the legitimate wininit.exe — that points to malware disguised under a trusted name.

Finishing the response

Eradication removes the root cause across every affected host — deleting the malware, disabling breached accounts, and fixing the exploited vulnerability everywhere it exists, not just on the first machine. Recovery restores normal operation from clean backups, then patches, rebuilds and rotates credentials. Lessons Learned, the phase most often skipped, reviews what happened, how well the team responded, and what to improve so the same incident does not recur.

Worked example — malware on a workstation

Endpoint protection and an Intrusion Detection System alert (both indicators) flag suspicious outbound traffic. You contain the incident by isolating the machine from the network. Using a write blocker, you capture memory first and then a disk image, and you hash both and log the handover. In the Volatility Framework, the process tree shows lsass.exe with the wrong parent process, confirming the malware. You then eradicate it from every affected host, recover by rebuilding from a clean backup and rotating credentials, and finish with a lessons-learned review. The chain-of-custody log records every step throughout.

Common mistakes

  • Confusing a precursor with an indicator.
  • Hashing the image only once instead of before and after handling.
  • Saying you would analyse the original drive — you always analyse the copy.
  • Using the wrong number of phases when the question names a specific framework.

Vulnerability assessment: risk & remediation

Module 05›

What this question tests. Whether you can run a vulnerability assessment properly, assess and prioritise the risks you find, and recommend layered remediation, then verify that it worked.

What a vulnerability assessment is, and is not

A clear distinction that often scores marks: a vulnerability assessment finds and ranks weaknesses across many systems and does not necessarily exploit them — it measures breadth. A penetration test goes further and proves that a weakness can actually be exploited on a target — it measures depth. The scenario usually wants the assess-and-recommend approach rather than full exploitation.

Credentialed versus non-credentialed scans

The choice changes the results, so explain why. In a credentialed (authenticated) assessment the scanner logs in to each machine, so it can see missing patches and configuration problems from the inside; this is deeper and more reliable and produces fewer false positives, but it comes at the cost of having to obtain and manage the credentials safely. In a non-credentialed (unauthenticated) assessment the scanner has no login and can only see the services that the host exposes to the network; this is quicker and gives an attacker's-eye view, but it is noisier, produces more false positives, and misses internal problems and anything hidden behind a firewall.

From findings to action

The work does not stop at running the scanner. First you validate the findings, because automated scanners produce false positives and you do not want the team spending effort patching something that is not really there. Then you prioritise by severity using the Common Vulnerability Scoring System (CVSS), together with the business impact, so that the critical remote-code-execution holes are fixed before the low-risk information leaks. Then you remediate in layers rather than relying on a single fix — patch or upgrade, harden the configuration, segment the network, apply least privilege, and tighten firewall rules. Finally you re-scan to verify that the issue is closed and then continue to monitor; this closing step completes the vulnerability-management lifecycle.

Worked example — a legacy Windows 2000 box

A credentialed scan flags the host. The finding is MS08-067 (CVE-2008-4250), a critical remote-code-execution vulnerability in Server Message Block. You validate that it is genuinely exploitable rather than a false positive, then prioritise it at the top of the list because it allows a full takeover. For remediation the ideal answer is to decommission the end-of-life operating system; if it must remain, you patch it, disable Server Message Block version 1, segment it away from the main network, apply least privilege, and apply tight firewall rules. You then re-scan to confirm the vulnerability is closed.

Common mistakes

  • Treating a vulnerability assessment as if it were a penetration test.
  • Skipping the validation step and the final re-scan.
  • Recommending only "apply the patch" — for an end-of-life system you score more by adding segmentation, least privilege and decommissioning.
