MOD 02

Footprinting & Reconnaissance

passive vs active · DNS/AXFR · social eng

Passive vs Active footprinting EXAM ★ ESSAY 1

Prep especially wants passive recon: gathering from public sources with no contact with the target.

Passive

No direct interaction → stealthy. OSINT, search engines, social media, job posts, Whois, DNS records, public repos (GitHub/Pastebin), document metadata.

Active

Directly touches the target → detectable. Port/network scanning, service fingerprinting, email harvesting by probing, website/dir enumeration.

Methodology order: identify target → passive → active → network info → social engineering → website → DNS → compile & report.

DNS zone transfer / AXFR EXAM ★

A secondary DNS server normally requests a full copy of the zone from the primary using an AXFR request (for redundancy). If the primary is misconfigured to allow anyone, an attacker mimics a secondary and pulls the entire zone file — every hostname, IP and record in one shot.

What the attacker gains: a full internal map — subdomains, mail servers, internal IPs, key hosts to target next.

# test for an open zone transfer
dig axfr @ns1.example.com example.com
nslookup -type=axfr example.com ns1.example.com

Social engineering — focus: phishing EXAM

Manipulating people into revealing info or creds — exploits human weakness, not a system bug.

Techniques

  • Phishing — fake emails harvesting credentials.
  • Impersonation — pose as IT/vendor.
  • Dumpster diving — secrets in the trash.
  • Shoulder surfing / eavesdropping.
  • Social-media profiling — habits & targets from LinkedIn/FB/IG.

Mitigation & limits

Mitigate: security-awareness training, a reporting culture, verification procedures, minimise public info. Limitation: phishing relies on human error — trained, skeptical staff and reporting defeat it; hard to target precisely at scale.

MOD 03

Computer & Network Scanning

★ Heavy zone — also Essay 2

TCP 3-way handshake & flags EXAM ★

Setup: SYN → SYN/ACK → ACK = connection established. Teardown: graceful FIN exchange, or abrupt RST.

FlagMeaning
SYNSynchronize — start a connection
ACKAcknowledge a received segment
FINFinish — graceful close
RSTReset — abort the connection
PSHPush — deliver data immediately
URGUrgent — prioritise this data

Host discovery: ARP vs ICMP EXAM ★ ESSAY 2

Finding live hosts (different question from finding open ports).

  • ICMP echo ping (-PE) — classic ping; simple but often blocked by firewalls.
  • ARP ping scan — Layer-2; most reliable on the local subnet, finds hosts even behind restrictive firewalls, and shows the MAC address. Can't cross routers.
  • UDP ping (-PU, default port 40125) — useful against hosts with strict TCP filtering.
# ping sweep + verify
nmap -sn 192.168.x.0/24 # host discovery only, no port scan
arp-scan --localnet

Port scanning techniques EXAM ESSAY 2

ScanNmapHow / why
TCP Connect (full open)-sTCompletes the full handshake → reliable but logged.
SYN / half-open (stealth)-sSSYN→SYN/ACK→RST (never finishes). Stealthier, needs root.
NULL / FIN / Xmas-sN -sF -sXInverse flag scans. Open port = no reply, closed = RST.
UDP-sUNo handshake; slow; result often open|filtered.
ACK-sAMaps firewall rules (stateful vs simple filter).

TRAP Inverse scans (NULL/FIN/Xmas): FIN=FIN only, NULL=no flags, Xmas=FIN+URG+PSH. Work on BSD-style stacks, not on Windows hosts.

Service/version detection & banner grabbing EXAM ★

nmap -sV fingerprints the service + version; nmap -O guesses the OS. Why it matters: a version string lets you map known CVEs straight to that service and pick a matching exploit.

Active banner grab

Send crafted/malformed packets; compare the TCP/IP-stack responses to a database.

Passive banner grab

Sniff existing traffic and infer the OS from telltale signs — no probing.

Scanning beyond IDS/firewall (evasion) EXAM ★ ESSAY 2

Packet fragmentation — split the probe into tiny fragments; many IDS skip fragmented packets (too costly to reassemble), and the host reassembles them. Tools: nmap -f, fragroute.

The rest of the evasion toolkit

  • IP decoys (-D) & IP spoofing — hide your real source among fakes.
  • Source-port manipulation — disguise as a trusted port (e.g. 53/80).
  • Source routing, custom packets, bad checksums.
  • Randomize host order; slow / timing scans to stay under detection thresholds.
  • Proxy servers / proxy chaining & anonymizers — bury the origin.

What enumeration is EXAM ESSAY 1·2

Extracting usernames, machine names, shares, services by making active connections and sending directed queries. This is where "port 445 is open" becomes "share X, user Y, version Z" — i.e. it turns scan results into the exact exploit to use.

Enumeration types — what each leaks EXAM ★

# the workhorses

Secure remote admin: SSH vs Telnet EXAM

SSH (22) is encrypted — the secure way to administer Linux. Telnet (23) sends everything, including credentials, in cleartext. Exam answer for "secure remote administration of Linux" = SSH.

MOD 06

System Hacking & Post-Exploitation

PrivEsc · persistence · C2 · covering tracks

Privilege escalation: horizontal vs vertical EXAM ★

Horizontal

Same privilege level, but access another peer user's stuff. E.g. banking user A reaches user B's account.

Vertical

Jump to a higher privilege — user → admin/root. E.g. a normal user reaching admin functions.

Gaining & maintaining access EXAM ESSAY 1

Gain access: crack passwords, exploit a buffer overflow, or exploit a known vulnerability (often via Metasploit). Maintain access (persistence): install backdoors / rootkits, hide files via NTFS ADS, keep a reverse shell.

Malware C2 behaviour EXAM

A compromised host beacons / calls back to an attacker-controlled server over a two-way channel — web traffic, email, or DNS — usually encrypted to hide it. Network anti-forensics channels: reverse HTTP shells, reverse ICMP tunnels, DNS tunneling.

Covering tracks / anti-forensics EXAM ★ ESSAY 1

  • Disable auditing, then clear or manipulate logs (delete only attack events to dodge detection).
  • Windows event logs: SECEVENT.EVT (security — failed logins, file access without rights), SYSEVENT.EVT (system — driver failures), APPEVENT.EVT (applications).
  • Tooling: clearev in a Meterpreter shell; wevtutil; PowerShell; the Clear_Event_Viewer_Logs.bat utility.
  • Hide on OS via NTFS ADS; delete unrecoverably with cipher.exe; disable last-access timestamps / restore points.

Authentication log analysis EXAM

Defender side: tell successful vs failed logins apart. A burst of failures then a success = brute force / password guessing. Watch for off-hours admin logins = privilege abuse.

Bonus Intel · Windows auth

Where passwords live

SAM database stores hashes (locked while Windows runs). NTLM = challenge/response; Kerberos = ticket-based via the KDC (AS + TGS), now the default — stronger than NTLM.

Bonus Intel · Password cracking

Attack types confirm

  • Dictionary
  • Brute-force
  • Hybrid
  • Rainbow tables (precomputed hashes)
Bonus Intel · Metasploit

Architecture

Core/Base APIs on top of the Rex library. Modules: exploits, payloads (meterpreter/shell), encoders (dodge AV), auxiliary (scanners), post, and a listener. Interfaces: msfconsole, msfvenom, Armitage.

MOD 07

Incident Response & Digital Forensics

IR life cycle · evidence · chain of custody

★ The 4-vs-6-phase trap

Your prep flags this. Three models float around — know which your question wants. Default to the NIST 4-phase (what the Module 07 lecture teaches); be ready to mention the others.

  • NIST 4-phase (lecture): Preparation → Detection & Analysis → Containment, Eradication & Recovery → Post-Incident Activity (Lessons Learned).
  • SANS 6-step: Preparation → Identification → Containment → Eradication → Recovery → Lessons Learned. (This is the one your group forensic assignment used — and Essay 3 mirrors that assignment.)
  • EC-Council IH&R 9-step (from Module 01) — the granular version.

IR goals & detection EXAM ESSAY 3

Goals: minimise damage, reduce recovery time, restore normal operations fast. Detection & Analysis is the hardest phase — "incidents can't always be prevented but must always be detected."

Precursors vs Indicators

Precursor

A sign an incident may happen — e.g. a vuln scanner in your web logs, a public exploit released for your mail server.

Indicator

A sign one has/ is happening — e.g. AV flags malware, IDS alerts on a successful DB attack.

Common attack vectors

External/removable media (USB), Attrition (brute force), Web, Email, Impersonation (spoofing/MITM/rogue AP/SQLi), Improper usage, Loss/theft of equipment.

Evidence gathering & handling EXAM ★ ESSAY 3

Keep a detailed log for every piece of evidence — this is the basis of chain of custody:

  • Identifying info: location, serial & model number, hostname, MAC & IP.
  • Name, title & phone of everyone who collected or handled it.
  • Time & date (with time zone) of every handling event.
  • Where it was stored (tamper-evident, secure).

Containment strategy is chosen on: potential damage, need to preserve evidence, service availability, time/resources, and effectiveness.

Eradication, recovery & lessons learned EXAM

Eradicate: delete malware, disable breached accounts, fix exploited vulns across all affected hosts. Recover: restore from clean backups, rebuild, patch, change passwords, tighten the perimeter. Lessons learned (most often skipped): what happened, when, how well the team did, what to watch for next time.

MOD 08

Memory Acquisition & Analysis

imaging · hashing · Volatility

Forensics process & hashing EXAM ★ ESSAY 3

Process: Acquisition → Analysis → Documentation/Presentation. Goals: what happened, how, who, when, where, why.

Imaging = a bit-for-bit forensic copy (use a write blocker; work on the copy, preserve the original). Hashing proves integrity: take an MD5/SHA-256 hash before and after — if the hash is unchanged, the evidence is unaltered.

TRAP Hashing = integrity, not confidentiality, and it's one-way (can't be reversed).

Why memory is gold EXAM

Volatile RAM holds things the disk never sees: running processes, executed commands, cleartext passwords, unencrypted data, open network connections, IPs, injected trojans. Capture it before powering off.

Volatility workflow ESSAY 3

# identify, then list, then hunt
vol.py -f mem.raw imageinfo
vol.py -f mem.raw --profile=<P> pslist / pstree / psscan
vol.py -f mem.raw --profile=<P> psxview # find hidden processes
vol.py -f mem.raw --profile=<P> netscan # network connections

IoC A great red flag: lsass.exe's parent should be wininit.exe. If its parent is explorer.exe, it's a fake process wearing a legit name.

MOD 10

Cryptography Bonus · light in prep

symmetric · asymmetric · hashing

Your prep only touches crypto through hashing for evidence integrity (covered in Module 08). The rest is here as MCQ insurance — if your lecturer confirmed crypto is out of scope, hit "Exam targets only" to hide it.

Symmetric vs asymmetric

Symmetric (private-key)

Same key encrypts & decrypts. Fast, for bulk data. DES, AES, RC4, IDEA.

Asymmetric (public-key)

Two keys (public + private). RSA, Diffie-Hellman, El Gamal, DSA.

Block vs stream: block ciphers work on fixed-size blocks (AES, DES — most modern ciphers); stream ciphers encrypt bit-by-bit with a keystream (RC4, SEAL).

Hashing (one-way) links Essay 3

MD5, SHA-1, SHA-2 produce a fixed-size digest that can't be reversed. Used to prove a file/evidence hasn't changed. The 4 crypto objectives: confidentiality, integrity, authentication, non-repudiation.

Bonus · Diffie-Hellman

Key exchange over an open line

Agree on public primes n, g; each side picks a secret and computes A=g^x mod n / B=g^y mod n; both derive the same shared key. Lab note: it's vulnerable to a man-in-the-middle attack — that's a likely essay/lab question.

MOD 09

Steganography Bonus · not in prep

covert channels · LSB · ADS

Not flagged anywhere in your prep — pure bonus. Worth a skim for MCQs (it also connects to "hiding files" in Module 06).

The core idea

Steganography = hiding the very existence of a message (vs cryptography, which hides the meaning). It's one kind of covert channel — moving data between parties who aren't supposed to communicate, slipping past firewalls/IDS.

Types: image, audio, text, email, and whitespace (trailing spaces/tabs).

LSB & ADS

Least Significant Bit — swap the rightmost bit of each byte/pixel for your secret bits; the image looks unchanged. Alternate Data Streams (NTFS) — hide data in a second stream: normal.txt:hidden.txt doesn't show in Explorer.

Fun fact · history

Old-school hiding

Demaratus hid a message under the wax of a tablet; WWII used invisible ink and the null cipher — read the first letter of each word: "Please Do Not Throw Sausage Pizza Away."

Fun fact · malware

Stegoloader

A real trojan that hid its main module inside a PNG image downloaded from a legit site to dodge detection.

Bonus · detection

File signatures & tools

Magic numbers (file header) reveal the true type even if the extension lies. Tools: TrID, Binwalk, file; stego tools: Steghide, QuickStego, MP3Stegz, SNOW.

Section B · where the marks are

The 4 scenario essays — deep battle plans

This is the hard half, so each one below now has four parts: what the question is really testing, the concepts you must actually understand, a full-marks answer structure, and a worked example. Tap to open. Golden rule: every point must be tied back to the scenario you're given — generic theory with no link to "the company in the question" loses marks.

1

Attack life cycle & methodology

Modules 02·03·04·06 · usually 3 sub-parts›

What's being tested: can you walk a real attacker's path end-to-end against the org in the scenario, name the correct technique at each stage, explain why each stage feeds the next, and pair the attack with a defence. It's the Cyber Kill Chain / 5 hacking phases applied to a story, not recited as a list.

The idea behind it — why the chain is a chain

The single biggest mark-winner is showing the cause-and-effect link between stages, because that's what proves you understand the methodology rather than memorising six words:

  • Recon → Scanning: you can't scan until recon has given you the IP ranges / domains / which hosts exist.
  • Scanning → Enumeration: an open port (445) is just a door; enumeration is what tells you the share name, the username, the exact version behind it.
  • Enumeration → Gaining access: that version string is what you match to a CVE and a specific exploit. Without it you're guessing.
  • Access → Maintaining access: a shell is lost on reboot, so you escalate privilege and drop a backdoor / rootkit to stay in.
  • → Clearing tracks: finally you wipe logs and hide artefacts so the defender can't reconstruct what happened.

Map this to the two models so you can name-drop either: the CEH 5 phases (Recon → Scanning → Gaining → Maintaining → Clearing) and the Lockheed Martin 7-stage kill chain (Recon → Weaponization → Delivery → Exploitation → Installation → C2 → Actions on Objectives) are the same story at two levels of detail.

How to structure a full-marks answer
  1. Frame it. One line naming the model and the target from the scenario: "I'll follow the CEH attack lifecycle against [the company]'s public-facing systems."
  2. Reconnaissance. Passive first (Whois, DNS records, crt.sh subdomains, Google dorks, theHarvester for emails, LinkedIn/job posts, document metadata) — no contact, stealthy. Then active (probing, network mapping).
  3. Scanning. Host discovery (nmap -sn, ARP/ICMP ping sweep) → port & service scan (nmap -sV, -O for OS). State what you're looking for.
  4. Enumeration. Pull users/shares/versions from the open services (NetBIOS/SMB, SNMP, LDAP). Turn "445 open" into "SMBv1, Win2000, share X".
  5. Gaining access. Map the version → CVE → exploit. Name the route (e.g. Metasploit module, or a crafted msfvenom payload delivered by phishing).
  6. Maintaining access. Privilege escalation (vertical to admin/SYSTEM), then persistence — backdoor, rootkit, scheduled task, hide via NTFS ADS.
  7. Clearing tracks. Disable auditing, clear/selectively edit event logs (clearev), timestomp, cipher.exe to wipe.
  8. Defences (if asked, or to score extra). Pair each stage with a control — see the Module 01 kill-chain table. This is often a whole sub-part.
Worked example — a legacy-systems company (the classic scenario)

Recon: Whois + crt.sh reveal mail.acme-corp.com and an old vpn.acme-corp.com; a job ad mentions "Windows Server 2000". Scan: nmap -sV on that host shows 445/tcp open, microsoft-ds, Windows 2000. Enumerate: a null IPC$ session lists shares and users. Gain access: Win2000 SMB → MS08-067 (CVE-2008-4250), a critical remote code execution — fire the Metasploit module, get a Meterpreter session. Maintain: migrate into a stable process, add a backdoor service. Clear: clearev + wipe the dropped files. Defence: patch / decommission the EOL OS, segment it off the main LAN, disable SMBv1, least privilege, IDS/IPS, email filtering + awareness for the phishing route.

Mark-grabbers: name the model · keep referencing "the company in the scenario" · pair every attack step with a countermeasure · use correct tool + CVE names · and one easy point most people forget — note that as an ethical hacker you'd only do this within an authorised scope / signed agreement.

Common slip

Listing the phases generically with no link to the scenario, and forgetting the defence half when the question says "…and recommend how the organisation could prevent it." Read every sub-part.

Scanning, host/port discovery & IDS evasion

Module 03 (+04)›

What's being tested: do you understand the difference between "which hosts are alive" and "which ports/services are open", can you explain the packets behind each scan, and can you explain how unusual / crafted packets slip past an IDS or firewall. Expect 3–4 sub-parts.

The ideas behind it
Host discovery

Answers "which machines are up?" — ICMP echo ping (often blocked), or ARP scan (Layer 2, only works on the local subnet but very reliable and shows the MAC), or UDP ping for strict TCP filtering.

Port / service discovery

Answers "which doors are open and what's behind them?" — TCP/UDP scans to find open ports, then -sV to fingerprint the service + version.

Then the why behind the key scans — this is what earns marks over a bare definition:

  • Why SYN / half-open (-sS) is "stealth": it sends SYN, gets SYN/ACK, then replies RST instead of ACK — the connection is never completed, so older loggers that only record established connections never see it. Connect scan (-sT) finishes the full handshake → it gets logged.
  • Why NULL / FIN / Xmas work (and where they don't): by RFC, a closed port replies RST and an open port stays silent to weird flag combos. So "no reply = open". They work on BSD-style TCP/IP stacks but not on Windows, which replies RST regardless.
  • Why -sV matters: a version string ("vsftpd 2.3.4") is the bridge to a known CVE and a matching exploit — it's the whole point of scanning before attacking.
Why crafted packets evade an IDS/firewall (explain the mechanism, don't just list)
  • Packet fragmentation (nmap -f) — splits the probe into tiny fragments. Many IDS won't pay the cost to reassemble them, so the signature never matches; the target host reassembles them normally.
  • Decoys (-D) & IP spoofing — bury your real source IP among many fakes so the analyst can't tell who's really scanning.
  • Slow / timing scans — stretch probes over time to stay under the IDS's rate threshold for "this looks like a scan".
  • Source-port manipulation — send from a trusted port like 53 or 80 to abuse loose firewall rules that allow that traffic.
  • Proxies / proxy chaining / anonymizers — route through intermediaries so logs point at them, not you.
How to structure a full-marks answer
  1. Separate the two goals up front: live-host discovery vs open-port/service discovery — examiners love this distinction.
  2. Host discovery method for the scenario (ICMP blocked by their firewall? → use ARP on-subnet) with the reasoning.
  3. Port-scan technique + the packet exchange; pick the stealthy one if they're trying to avoid detection.
  4. Service/OS detection (-sV, -O) and why the version is the key to exploitation.
  5. How enumeration converts results into a specific attack ("445 open" → share/user/version → exact exploit).
  6. IDS/IPS evasion — pick 2–3 techniques and explain why each one defeats detection.
Worked example — a firewalled network that drops ICMP

ICMP ping returns nothing, so I'd run an ARP scan on the local subnet (arp-scan --localnet) to find live hosts reliably. On each host, a SYN scan (nmap -sS) for stealth, plus -sV to fingerprint services. To get past their IDS I'd combine fragmentation (-f) with a couple of decoys (-D) and a slow timing template — fragments dodge signature matching, decoys hide my IP, slow timing stays under the scan-rate alarm. The open services + versions then feed enumeration to pick the exploit.

Mark-grabbers: the live-host-vs-open-port distinction · correct nmap flags · the actual packet exchange for SYN/Connect · a real reason for each evasion technique.

Common slips

Mixing up host discovery with port scanning · forgetting that NULL/FIN/Xmas fail on Windows · and remember evasion is taught in Module 03 (scanning), not the enumeration lab.

Incident response & digital evidence handling

Modules 07 & 08 · mirrors your group assignment›

What's being tested: can you run an incident end-to-end with a recognised IR lifecycle, AND handle digital evidence so it stays court-admissible — imaging, write blocker, order of volatility, hashing, chain of custody. This is the one that mirrors your group forensic assignment, so lean on what you did there.

Concept 1 — pick a lifecycle, and know they're the same shape

Don't panic about 4 vs 6 vs 9. They're the same process at different resolutions — showing you know that is itself a mark:

NIST 4-phase (lecture default)

Preparation → Detection & Analysis → Containment, Eradication & Recovery → Post-Incident Activity.

SANS 6-step (your assignment)

Preparation → Identification → Containment → Eradication → Recovery → Lessons Learned.

SANS just splits NIST's middle phase into separate Containment / Eradication / Recovery steps and pulls "Identification" out of Detection. EC-Council's 9-step IH&R is the same idea broken down further (Prep → Recording → Triage → Notification → Containment → Evidence Gathering → Eradication → Recovery → Post-Incident). Use NIST 4 as your spine unless the question names one.

Why Detection & Analysis is the hard phase: you have to spot the incident from weak signals. Know the pair:

Precursor

A sign an incident may happen — a vuln scanner showing up in your web logs, a new public exploit for software you run.

Indicator

A sign one is / has happened — AV flags malware, IDS alerts on a successful DB attack, a file-integrity tool sees a changed system file.

Concept 2 — evidence integrity (the highest-value cluster)

This trio is what makes evidence stand up in court — explain the why of each, not just the name:

  • Bit-for-bit image + write blocker: you take an exact forensic copy (which captures slack space and deleted data, not just visible files) and always work on the copy. The write blocker physically prevents any write to the original drive — a single write changes the disk and the defence can argue the evidence was altered.
  • Order of volatility: capture the most volatile data first — RAM before you power off — because memory holds running processes, executed commands, cleartext passwords, live network connections and injected malware that all vanish at shutdown.
  • Hashing (MD5 / SHA-256): hash the image before and after handling. If the two hashes match, you've mathematically proven the data wasn't changed — that's integrity. (It's one-way and not encryption.)
  • Chain of custody: a documented log of who handled what, when (with time zone), and where for every handoff, plus secure tamper-evident storage. It removes any "someone could have tampered with it" defence.
How to structure a full-marks answer
  1. State the lifecycle you'll use (NIST 4; mention SANS 6 from your assignment) and the incident from the scenario.
  2. Preparation — the team, tools and policies that were ready beforehand.
  3. Detection & Analysis — what tipped you off (precursor vs indicator), then triage: scope, severity, affected systems.
  4. Containment — isolate the host (pull from network); note the criteria for choosing a strategy (damage, evidence preservation, availability, time, effectiveness).
  5. Evidence acquisition — write blocker, bit-for-bit image, capture RAM first (order of volatility), hash before/after.
  6. Chain of custody — the log contents (IDs, MAC/IP, who/when/where, storage).
  7. Eradication & Recovery — remove malware/root cause across all affected hosts, then restore from clean backups, patch, change passwords.
  8. Post-incident / lessons learned — what happened, how the team did, what to improve.
  9. If a memory image is involved — Volatility: imageinfo → pslist/pstree → psxview → netscan, and the IoC below.
Worked example — malware/ransomware on a workstation

Detect: EDR + an IDS alert (indicators) flag suspicious outbound traffic. Contain: isolate the machine from the network immediately. Acquire: with a write blocker, capture RAM first then a disk image; hash both and log it. Analyse: in Volatility, pstree shows lsass.exe with the wrong parent — a fake process. Eradicate: remove the malware and close the entry vector on every affected host. Recover: rebuild from a clean backup, patch, rotate credentials. Lessons learned: document the timeline and tighten the gap that let it in. Throughout, the chain-of-custody log records every handoff.

Mark-grabbers: name a recognised model · the write-blocker + hashing + chain-of-custody trio is the single richest cluster of marks · capture RAM before disk · tie every step to the scenario.

Common slips

Confusing precursor vs indicator · forgetting to hash both before and after · saying you'd analyse the original (always the copy) · and picking the wrong phase count when the question explicitly names NIST or SANS.

Vulnerability assessment: risk & remediation

Module 05›

What's being tested: can you run a VA properly (scope → tool → credentialed-or-not → validate), assess and prioritise the risk you find (CVSS), and recommend layered remediation — then close the loop by rescanning. Usually 3 sub-parts: conduct / identify risk / remediate.

Concept 1 — VA is not a pen test

A clean distinction that often scores: a vulnerability assessment finds and ranks weaknesses across many systems (breadth) and doesn't necessarily exploit them; a penetration test proves exploitability on a target (depth). The scenario will usually want the assessment-and-recommend angle.

Concept 2 — credentialed vs non-credentialed, and why it changes the result
Credentialed (authenticated)

The scanner logs in, so it sees missing patches and config from the inside — deeper and more reliable, fewer false positives. Cost: you have to manage credentials safely.

Non-credentialed

No login — only sees what's exposed to the network. Faster and good for an attacker's-eye view, but noisier, more false positives, and misses internal/host-level issues.

Why validation matters: automated scanners throw false positives. You confirm findings before reporting so the team doesn't burn effort patching something that isn't really there — that's the step that separates a real assessment from "I ran a tool".

Why prioritise by CVSS: you can't fix everything at once, so you rank by severity (CVSS score) and business impact and fix the critical remote-code-execution holes before the low-risk info leaks.

How to structure a full-marks answer
  1. Scope & assets. Define what's in scope and baseline the systems (IPs, OS, services).
  2. Pick the tool + scan type. Nessus / OpenVAS / Qualys (or Metasploit + Nmap as in your lab); choose credentialed vs non-credentialed and justify it.
  3. Run the scan and collect findings.
  4. Validate the results to cut false positives.
  5. Classify & prioritise by CVSS severity (+ business impact); name CVE/CWE where relevant.
  6. Remediate — layered: patch/upgrade, harden config, segment, least privilege, firewall rules. Not just "install the patch".
  7. Rescan to verify, then monitor continuously. This closes the VA lifecycle loop.
Worked example — legacy Windows 2000 box on the network

Scope/scan: a credentialed scan with Nessus (or Nmap + Metasploit in the lab) flags the host. Finding: MS08-067 (CVE-2008-4250) — a critical remote-code-execution vulnerability in SMB. Validate: confirm it's genuinely exploitable, not a false positive. Prioritise: critical RCE → top of the list, since it gives full takeover. Remediate: ideally decommission the end-of-life OS; if it must stay, patch, disable SMBv1, segment it off the main network, apply least privilege and tight firewall rules. Verify: rescan to confirm it's closed.

Mark-grabbers: the VA-vs-pen-test line · the credentialed/non-credentialed trade-off · CVSS prioritisation · and the often-missed rescan to verify that closes the loop.

Common slips

Treating VA as a pen test · skipping the validation step and the final rescan · and recommending only "patch it" — for an EOL system you score more by adding segmentation, least privilege and decommissioning.

Hashing gives integrity and is one-way. Encryption gives confidentiality and is reversible with the key.
